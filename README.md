Ты работаешь с полным актуальным репозиторием Polymarket-бота. В репозитории нет мусорных копий и старых версий: считай, что это чистая живая версия проекта.

Твоя задача — провести глубокий технический аудит архитектуры, взаимодействия файлов и логики исполнения, а затем внести точечные и безопасные изменения в код только там, где это действительно нужно.

ГЛАВНАЯ ЦЕЛЬ:
Сделать максимально чистую, устойчивую и предсказуемую архитектуру без лишнего кода, дублирования и скрытых багов, но при этом СОХРАНИТЬ ТЕКУЩЕЕ ПОВЕДЕНИЕ системы. Это критично. Нельзя “улучшать” стратегию, если это меняет фактическую логику принятия решений. Сначала надежность, чистота и консистентность. Потом красота.

КОНТЕКСТ:
Проект — это бот с архитектурой примерно такого типа:

* collector / ingestion
* filters
* scorer
* regime detector
* edge registry
* main orchestrator
* paper engine / portfolio simulation
* state persistence / runtime memory

Известные зоны риска, которые нужно проверить особенно внимательно:

1. shadowing / дублирование логики между main и отдельными модулями
2. реальное взаимодействие между файлами, а не только формальные import’ы
3. сохранение и восстановление runtime state
4. консистентность PaperEngine после рестарта
5. герметичность candidate pipeline
6. раздутые orchestration-функции
7. раздутые evaluation-функции в paper engine
8. скрытое дублирование порогов, условий, gate-логики и state-флагов
9. неиспользуемый код, мертвые ветки, мертвые helpers, фиктивная модульность
10. места, где код “работает”, но архитектурно уже хрупкий и склонный к регрессам

ЧТО НУЖНО СДЕЛАТЬ:

1. Построить карту зависимостей проекта:

   * какие файлы реально участвуют в runtime
   * кто источник истины для regime logic
   * кто источник истины для portfolio / paper state
   * где логика размазана по нескольким файлам

2. Найти и устранить архитектурные дефекты без изменения фактической стратегии:

   * убрать shadowing логики
   * убрать фиктивную модульность
   * убрать опасное дублирование логики
   * сделать поведение более детерминированным

3. Проверить persistence:

   * сохраняется ли весь необходимый runtime state
   * восстанавливается ли PaperEngine полностью
   * не возникает ли разрыва между state memory и portfolio state после рестарта

4. Проверить candidate pipeline:

   * нет ли повторного впуска сырого мусора после фильтрации
   * нет ли обхода санитарных фильтров через боковые ветки
   * нет ли архитектурной утечки, где сначала фильтруем, потом снова используем raw candidates

5. Проверить orchestration:

   * разбить чрезмерно жирные функции на логические части
   * не ломая итоговый порядок решений
   * оставить последовательность принятия решений максимально эквивалентной текущей

6. Проверить PaperEngine:

   * разбить раздутые evaluation / lifecycle-блоки на более мелкие функции
   * не менять фактические exit / hold / rotate / cooldown / lock semantics
   * сделать код более проверяемым и читабельным

7. Провести cleanup:

   * удалить реально неиспользуемый код
   * удалить мертвые helpers
   * удалить фиктивные compatibility-слои, если они не нужны
   * убрать лишние промежуточные флаги, если они ничего не дают
   * но только если точно доказано, что это безопасно

ЧТО ДЕЛАТЬ НЕЛЬЗЯ:

1. Нельзя менять стратегию торговли.
2. Нельзя менять поведение scoring, filtering, admission, risk-routing, exits, political overrides, family logic и competition logic, если это не исправление явного бага.
3. Нельзя “упрощать” код ценой потери логики.
4. Нельзя делать косметический рефактор, который визуально красивый, но ухудшает проверяемость.
5. Нельзя удалять seemingly redundant код без доказательства, что он реально не используется.
6. Нельзя менять имена, пути и структуру файлов без реальной необходимости.
7. Нельзя вносить speculative improvements “на будущее”, если они меняют текущую механику.
8. Нельзя молча ломать совместимость запуска через текущую entrypoint-схему.

ЧТО НАДО ДЕЛАТЬ:

1. Сначала аудит, потом изменения.
2. Сначала доказать проблему, потом фиксить.
3. Все изменения делать минимально инвазивно.
4. Сохранять API между файлами, если нет сильной причины менять.
5. Если создаешь новый файл — только если это реально улучшает архитектуру и уменьшает хрупкость.
6. Любой рефактор должен сопровождаться проверкой:

   * проект компилируется
   * imports целые
   * entrypoint работает
   * state restore не сломан
   * behavior максимально эквивалентен предыдущему

ФОРМАТ РАБОТЫ:

1. Сначала выдай короткий аудит:

   * что найдено
   * что опасно
   * что точно нужно исправить
   * что лучше не трогать

2. Затем внеси правки в код.

3. После правок выдай отчет:

   * какие файлы изменены
   * какие новые файлы добавлены
   * какие архитектурные проблемы устранены
   * что намеренно НЕ менялось, чтобы не нарушить поведение
   * какие потенциальные риски все еще остаются

КРИТЕРИЙ УСПЕХА:
После твоей работы код должен быть:

* чище
* короче там, где это безопасно
* без фиктивной модульности
* без лишнего дублирования
* с настоящей persistence там, где она нужна
* с более прозрачным orchestration flow
* с максимально сохраненным текущим поведением системы

Если придется выбирать между “сделать красивее” и “сохранить поведение”, всегда выбирай сохранение поведения.

Работай как инженер-аудитор, а не как художник-реформатор.
